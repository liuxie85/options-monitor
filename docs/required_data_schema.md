# required_data schema (options-monitor)

This project consumes per-symbol option-chain snapshots generated by:
- `scripts/fetch_market_data.py`

Outputs:
- Raw JSON: `output/raw/{SYMBOL}_required_data.json`
- Parsed CSV: `output/parsed/{SYMBOL}_required_data.csv`

These CSV files are consumed by:
- `scripts/scan_sell_put.py`
- `scripts/scan_sell_call.py`

This document defines the **minimum required columns** and the current conventions.

## Minimal contract row schema

Each row represents **one option contract**.

### Identity / dimensions
- `symbol` (str): underlying ticker (e.g. `NVDA`)
- `option_type` (str): `put` or `call`
- `expiration` (str): `YYYY-MM-DD`
- `dte` (int): days to expiration
- `contract_symbol` (str): current identifier (from yfinance `contractSymbol`).

### Pricing inputs (required by scanners)
- `spot` (float): underlying spot price
- `strike` (float)
- `bid` (float | empty)
- `ask` (float | empty)
- `mid` (float | empty):
  - preferred: `(bid+ask)/2` when bid/ask are both > 0
  - fallback: `last_price` when positive
- `last_price` (float | empty): used only as `mid` fallback

### Liquidity / data-quality gates
- `open_interest` (float | empty)
- `volume` (float | empty)
- `implied_volatility` (float | empty):
  - convention: decimal (e.g. `0.50` for 50%)
  - some sources may output percent; scanners normalize heuristically when gating is enabled.

### Greeks
- `delta` (float | empty):
  - currently computed by `fetch_market_data.py` via Black–Scholes using IV (no scipy).
  - scanners may optionally require delta (delta gating flags).

### Optional / helpful
- `in_the_money` (bool | empty)
- `currency` (str): currently `USD`
- `otm_pct` (float | empty): computed from spot & strike; used for labeling/inspection.

## Scanner dependency map (high level)

- `scan_sell_put.py` requires: `dte, strike, spot, mid` and uses `open_interest, volume, bid, ask, implied_volatility, delta` for gating.
- `scan_sell_call.py` requires: `dte, strike, spot, mid` and uses `open_interest, volume, bid, ask, implied_volatility, delta` for gating.

## OpenD integration rule

When integrating additional sources (e.g. Futu OpenD), we should **preserve these columns**.
- Existing columns must not be removed/renamed.
- New columns may be added (e.g. `opend_code`, `gamma/theta/vega`, `source`, `elapsed_ms`).
- If OpenD provides greeks, prefer using OpenD values while keeping fallbacks to maintain coverage.
