# 期权通知 Markdown 全面优化方案

## 现状诊断

### 当前通知结构（完整链路）

```
# 📊 Options Monitor
## 账户提醒（lx）
北京时间 2026-05-12 22:31:00

### 账户 lx · 本轮候选
- Put 2 / Call 1 / Enhance 1 / 优化器 换仓1 平仓0

Put:
- 腾讯 卖Put 2026-04-29 460P
担保 1张 加仓后余量 ¥-100 【现金不足】

Call:
- 英伟达 卖Call 2026-06-18 180C
覆盖 1张 cover 1 【可覆盖】

变化:
英伟达 收益增强 2026-06-19 95P+110C

### [lx] 平仓建议
- NVDA Put 2026-06-19 150P · 强烈建议平仓换仓 🔄
- 已锁定: 85.0% | 剩余DTE=14 | 剩余收益年化=5.0%
- 价格: 开仓权利金=2.5 | 平仓 mid=0.4
- 估算: 平仓后锁定收益 $210 | 剩余权利金 $40
- 理由: switch
- 优化器: 持有年化=5.0% → 替代候选年化=12.0% | 尾部风险=0.045
---

💰 现金 CNY
LX 持有 ¥1,000 (CNY) | 可用 ¥200 (CNY)
```

### 跨文件渲染链路

| 阶段 | 文件 | 输出 | 当前问题 |
|------|------|------|----------|
| 候选渲染 | `render_sell_put_alerts.py` | `sell_put_alerts.txt` | 标题 `[Sell Put 候选]` 冗长，信息堆砌 |
| 候选渲染 | `render_sell_call_alerts.py` | `sell_call_alerts.txt` | 同上的问题 |
| 候选渲染 | `render_yield_enhancement_alerts.py` | `yield_enhancement_alerts.txt` | 同上的问题 |
| 告警合并 | `alert_engine.py` | `symbols_alerts.txt` | pipe 分隔符在 notify_symbols 里被重新解析 |
| 通知构建 | `notify_symbols.py` | `symbols_notification.txt` | `_build_notification_block` 生成大量 `- ` 行，密度高 |
| 平仓建议 | `close_advice_runner.py` | `close_advice.txt` | 每持仓 7 行，`|` 分隔符多，无视觉优先级 |
| 多账户包装 | `multi_tick/notify_format.py` | Feishu 消息 | 纯文本包装，无色彩/层级优化 |

### 核心问题汇总

1. **信息密度过高**：每个候选/持仓平均 5-8 行，`|` 分隔符在移动端难以扫描
2. **视觉层级扁平**：所有行都是 `- ` 开头，无法一眼区分"这是什么"和"我该做什么"
3. **标题冗长**：`[Sell Put 候选]`、`### [lx] 平仓建议` 占用空间且无 emoji 标识
4. **分隔线干扰**：`---` 在 Markdown 是水平线，占用一屏空间
5. **术语不友好**：`switch`、`DTE`、`OTM` 对运营人员有理解门槛
6. **数字格式不统一**：`%`、`$`、`¥`、纯数字混排
7. **候选与变化区段分离**：`Put:` / `Call:` / `变化:` 是平面列表，无动作导向

## 设计原则

1. **行动优先（Action-First）**：首行必须让用户知道"这是什么机会/风险"和"建议做什么"
2. **移动端优先**：每行不超过 30 个汉字等效宽度，适合手机竖屏
3. **视觉层级**：用 emoji + 缩进 + 空行建立 3 级信息层级（L1 行动 → L2 关键指标 → L3 详情）
4. **向后兼容**：通过 `notification.render_style` 配置开关控制，默认启用新样式
5. **Markdown-friendly**：只使用基础语法（`#`、`- `、`**`、emoji），兼容 Feishu 纯文本
6. **统一语言**：所有模块使用同一套 emoji 映射、数字格式、日期格式

## 方案：统一卡片式通知（Unified Card Notification）

### 1. 整体消息结构

```

## 账户提醒（lx）
⏰ 北京时间 2026-05-12 22:31

📋 本轮概览
  Put 2 · Call 1 · 增强 1
  🔴 优化器 换仓 1 · 平仓 0

─────────────────────

🔵 开仓候选

🟢 卖Put 腾讯 460P @ 04-29
  权利金 ¥2,300 · 年化 12%
  担保占用 ¥46,000 · 加仓后余量 🔴¥-100

🟢 卖Put 阿里 85P @ 05-17
  权利金 $180 · 年化 8%
  担保占用 $8,500 · 加仓后余量 🟢¥3,200

🟡 卖Call 英伟达 180C @ 06-18
  权利金 $120 · 年化 6%
  覆盖 100 股 · 可用 🟢200 股

  PutΔ=0.25 · Call ask=$1.2

### 2. 信息层级规范（跨模块统一）

#### L1 行动行（首行）
- 格式：`{emoji} {动作} {标的} {类型} {strike}{P/C} @ {日期}`
- 目的：一眼识别"这是什么"和"建议做什么"
- 长度限制：≤ 30 个字符等效宽度

#### L2 核心指标行（2 空格缩进）
- 格式：`  {指标1} · {指标2} · {指标3}`
- 目的：给出量化决策依据（收益、时间、风险）
- 数量限制：每持仓最多 2 行 L2

#### L3 详情行（2 空格缩进 + 前缀）
- 格式：`  💡 {补充信息}` 或 `  📊 {数据依据}`
- 目的：支撑 L1/L2 的深层数据
- 仅在有 optimizer / enhancement 时显示

### 3. Emoji 映射表（全局统一）

#### 区段标识

| 区段 | Emoji | 说明 |
|------|-------|------|
| 开仓候选总标题 | 🔵 | 新的交易机会 |
| 平仓建议总标题 | 🔴 | 需要处理的持仓 |
| 资金概览 | 💰 | 账户资金状态 |
| 收益增强 | 💎 | 组合策略机会 |
| 变化/更新 | 📝 | 已有持仓的状态变化 |

#### 单持仓动作标识

| 动作 | Emoji | 使用场景 |
|------|-------|----------|
| 🟢 | 卖Put/卖Call 候选 | 可开仓的新机会 |
| 🟡 | 卖Call（覆盖紧张） | 可开仓但覆盖不足 |
| 🔴 | 优化器 switch/close | 强烈建议行动 |
| 🟠 | 优化器 close | 建议平仓 |
| ⚪ | optional/weak | 可选关注 |
| 💎 | 收益增强 | 组合策略 |

#### 状态标识（行尾/内联）

| 状态 | 标识 | 使用场景 |
|------|------|----------|
| 现金不足 | 🔴 | 加仓后余量 < 0 |
| 现金支持 | 🟢 | 加仓后余量 ≥ 0 |
| 可覆盖 | 🟢 | 卖Call 有足量股票 |
| 不可覆盖 | 🔴 | 卖Call 覆盖不足 |
| 换仓建议 | 🔄 | 优化器 switch |
| 平仓建议 | ⚠️ | 优化器 close |

### 4. 数字格式化规范（全局统一）

| 类型 | 格式 | 示例 |
|------|------|------|
| 百分比 | ≥10% 显示整数，<10% 显示一位小数 | `12%`、`8.5%` |
| 货币（CNY） | `¥{N}`，去掉小数点后无意义零 | `¥2,300`、`¥46,000` |
| 货币（USD） | `${N}`，同上 | `$180`、`$8,500` |
| 天数 | `{N} 天` | `14 天`、`21 天` |
| 日期（当年） | `MM-DD` | `@ 04-29`、`@ 06-19` |
| 日期（跨年） | `YYYY-MM-DD` | `@ 2026-04-29` |
| Delta | 两位小数 | `Δ=0.25` |
| 风险分数 | 三位小数 | `0.045` |
| 权利金 | 带货币符号 | `¥2,300`、`$180` |

### 5. 跨模块改造详情

#### 模块 A：notify_symbols.py（候选通知构建）

**当前 `_build_notification_block` 输出**：
```
### [lx] 腾讯 · 卖Put
- 腾讯 卖Put 2026-04-29 460P
- 收益: 权利金=2.3 | 年化 12% | 净收 2300
- 合约: 行权价=460 | 数量=1张(默认) | DTE=29
- 风控: 风险=保守 | delta=0.25 | IV=0.45
- 资金: 保证金占用=¥46,000
- 备注: 通过准入
---
```

**新 `_build_notification_block_compact` 输出**：
```
🟢 卖Put 腾讯 460P @ 04-29
  权利金 ¥2,300 · 年化 12% · 剩余 29 天
  担保 ¥46,000 · 加仓后余量 🔴¥-100
```

**改动点**：
- 新增 `_build_notification_block_compact` 函数
- 标题从 `### [acct] symbol · action` 改为 `🟢 action symbol strikeP/C @ date`
- 合并 5-6 行 `- ` 列表为 2 行缩进内容
- 移除 `---` 分隔线，改用空行分隔持仓
- 保留原有 `_build_notification_block` 作为 `legacy` 回退

#### 模块 B：close_advice_runner.py（平仓建议渲染）

**当前 `render_markdown` 输出**（每持仓）：
```
- NVDA Put 2026-06-19 150P · 强烈建议平仓换仓 🔄
- 已锁定: 85.0% | 剩余DTE=14 | 剩余收益年化=5.0%
- 价格: 开仓权利金=2.5 | 平仓 mid=0.4
- 估算: 平仓后锁定收益 $210 | 剩余权利金 $40
- 理由: switch
- 优化器: 持有年化=5.0% → 替代候选年化=12.0% | 尾部风险=0.045
---
```

**新 `render_markdown_compact` 输出**（每持仓）：
```
🔴 换仓 NVDA Put 150P @ 06-19
  已锁定 85% · 剩余 14 天
  平仓收益 +$210（剩余 $40）
  💡 持有 5% → 替代 12% · 风险 0.045
```

**改动点**：
- 新增 `render_markdown_compact` 函数
- 首行 emoji 根据 tier 映射（🔴 switch / 🟠 close / 🟢 hold）
- 动词替换技术术语（`换仓`/`平仓`/`持有` 替代 `switch`/`close`/`hold`）
- 合并 6 行为 3-4 行
- 移除 `---` 分隔线

#### 模块 C：multi_tick/notify_format.py（消息包装）

**当前 `build_account_message` 输出**：
```
# 📊 Options Monitor

北京时间 2026-05-12 22:31:00

```

**新 `build_account_message_compact` 输出**：
```
# 📊 Options Monitor
## 账户提醒（lx）
⏰ 北京时间 2026-05-12 22:31

📋 本轮概览
  Put 2 · Call 1 · 增强 1
  🔴 优化器 换仓 1 · 平仓 0

**改动点**：
- 时间行加 ⏰ emoji
- 统计行改为缩进列表，用 `·` 替代 `/`
- 区段之间用 `─────────────────────`（20 个 ─）分隔
- 资金概览加 💰 emoji 标题，内容缩进
- 各区段标题带 emoji（🔵 开仓候选 / 🔴 平仓建议 / 💰 资金概览）

#### 模块 D：render_sell_put/render_sell_call/render_yield_enhancement（候选渲染）

这些模块渲染的文本最终进入 `sell_put_alerts.txt` 等文件，**但这些文件不是直接发送到 Feishu 的**——它们被 `alert_engine.py` 解析并转换为 pipe-delimited 格式写入 `symbols_alerts.txt`，然后 `notify_symbols.py` 重新解析并构建最终通知。

**结论**：优化候选渲染的**优先级较低**，因为最终格式由 `notify_symbols.py` 控制。但如果要优化中间文件的可读性，可以同步调整。

### 6. 配置契约

```json
{
  "notifications": {
    "render_style": "compact",
    "sections": {
      "show_overview": true,
      "show_dividers": true,
      "show_emoji": true
    }
  }
}
```

- `render_style`: `"legacy"` | `"compact"`，默认 `"compact"`
- `show_overview`: 是否显示顶部概览区，默认 true
- `show_dividers`: 是否显示区段分隔线，默认 true
- `show_emoji`: 是否显示 emoji 标识，默认 true

### 7. 实现优先级

| 优先级 | 模块 | 影响 | 工作量 |
|--------|------|------|--------|
| P0 | `notify_symbols.py` | 候选通知格式，用户每天看 | 中 |
| P0 | `close_advice_runner.py` | 平仓建议格式，新增功能 | 中 |
| P1 | `multi_tick/notify_format.py` | 整体消息包装 | 小 |
| P2 | `render_sell_put/call/enhancement` | 中间文件可读性 | 低 |

### 8. 测试策略

1. **契约测试**：
   - `test_notify_symbols_compact.py`：验证 `_build_notification_block_compact` 输出格式
   - `test_close_advice_compact_render.py`：验证 `render_markdown_compact` 输出格式
   - `test_notify_format_compact.py`：验证 `build_account_message_compact` 输出格式

2. **回归测试**：
   - 原有 `legacy` 样式测试全部保留
   - 切换 `render_style: legacy` 时输出与之前完全一致

3. **端到端测试**：
   - 完整 tick 流程后，验证 `symbols_notification.txt` 和 `close_advice.txt` 符合新格式
   - Feishu 消息长度不超过 4000 字符（微信限制）

### 9. 风险评估

| 风险 | 概率 | 缓解 |
|------|------|------|
| Feishu 对 `─` 分隔线渲染异常 | 低 | `─` 是 Unicode 制表符，所有客户端支持；异常时可关闭 `show_dividers` |
| 行尾 emoji（🟢/🔴）被截断 | 低 | 状态标识放在行内（如 `余量 🔴¥-100`），不是纯行尾 |
| 信息压缩过度导致决策依据不足 | 低 | 原始数据在 CSV 中完整保留；通知只展示决策层信息 |
| 用户习惯旧格式 | 中 | `render_style: legacy` 一键回退 |
| emoji 在部分终端显示为方框 | 低 | 本项目目标平台是 Feishu/微信，均支持标准 emoji |
